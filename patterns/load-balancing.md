# Load Balancing

## Что это
Load Balancer (LB) — это промежуточный компонент, который принимает входящие запросы (HTTP, TCP и т.п.) и распределяет их между несколькими серверами‑бэкендами.

Балансировщик скрывает от клиента реальную топологию серверов и выступает в роли единой точки входа в систему.

## Какую проблему решает
- Распределение нагрузки между несколькими серверами.
- Высокую доступность: если один сервер упал, запросы уйдут на живые.
- Горизонтальное масштабирование: можно добавлять новые серверы без изменения клиентского кода.

## Когда применять
- Когда трафик выходит за пределы одного сервера.
- Когда важно, чтобы сервис не “лежал”, когда один инстанс падает.
- Когда есть несколько сервисов/зон (например, web‑серверы, compute‑серверы, БД‑реплики) и нужно направлять запросы по типу.

## Как работает
- Клиенты обращаются на IP/domain балансировщика (например, api.example.com).
- LB выбирает целевой сервер по алгоритму (Round Robin, Least Connections и т.п.) и проксирует запрос.
- После получения ответа сервера LB возвращает его клиенту.

## Плюсы
- Лучше использует ресурсы кластера (нагрузка не скапливается на одном сервере).
- Повышает доступность: падение одного сервера не убивает весь сервис.
- Добавляет гибкость масштабирования (можно динамически добавлять/убирать серверы).
## Минусы
- Балансировщик сам может стать узким местом и единой точкой отказа, если его не резервировать.
- При L7‑балансировке добавляется немного задержки и сложности (парсинг HTTP, SSL‑терминация).
- В некоторых сценариях (например, долго живущие соединения) приходится думать про sticky sessions и их последствия.

## Пример
1. Сценарий: сайт с 10 web‑серверами за балансировщиком.
2. Клиент делает GET /api/user → запрос идёт на api.example.com → LB выбирает server по round‑robin → server обрабатывает и возвращает ответ через LB.
3. Если один из серверов упал, LB перестаёт слать на него трафик, остальные 9 серверов продолжают работу.

На практике это часто реализуется через:
- Nginx / HAProxy как самописный LB.
- Облачные сервисы: AWS ELB, GCP Cloud Load Balancing, Azure Load Balancer.
## Типичные ошибки
- Полагаться на один балансировщик без репликации/кластера.
- Игнорировать health‑checks и продолжать слать запросы на “мертвые” серверы.
- Использовать sticky sessions без понимания, как они влияют на распределение нагрузки и масштабирование.
